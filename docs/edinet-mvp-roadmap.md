# EDINET分析機能 MVPロードマップ

## 方針

MVPでは、実装と運用が軽い `EDINET DB API` をデータ取得元として利用する。

公式EDINET APIからの直接取得、XBRL解析、データベース導入はMVP後に検討する。アプリ内部では取得元に依存しない共通データ形式を利用し、将来公式EDINET APIへ切り替えられる構成にする。

```text
EDINET DB API
  -> Pythonによる取得・集計
  -> frontend/data/financials.json
  -> Next.jsによる表示
```

## MVPのゴール

対象企業について、直近5年の有価証券報告書由来データを表示・比較できる状態をMVPとする。

表示対象は次のとおり。

- 売上高
- 営業利益
- 純利益
- 営業利益率
- ROE
- 営業キャッシュフロー
- フリーキャッシュフロー
- 前年比
- 3年CAGR
- 2社比較
- データの取得日時
- 元となった有価証券報告書へのリンク

  - 営業利益

  - 総資産
  - 自己資本
  - 営業キャッシュフロー
  - 投資キャッシュフロー
  - ROE
- 自前で計算する分析値を次に限定する
  - 売上高前年比
  - 営業利益率
  - フリーキャッシュフロー
  - 3年CAGR
- 画面用JSONのデータ形式を決める

### 完了条件

- 入力データと出力データの意味を説明できる
- 欠損値と数値のゼロを区別する方針が決まっている

## Phase 2: 1社分のデータ取得

### 作業

- EDINET DBのAPIキーを取得する
- APIキーを環境変数で管理する
- 企業検索からEDINETコードを特定する
- 1社分の財務時系列データを取得する
- APIレスポンスを加工せずローカルへ保存する
- 年度、会計基準、欠損値、単位を確認する
- EDINET DBのWeb画面と取得値を照合する

### 完了条件

- 1社について直近5年分を取得できる
- APIの元レスポンスがローカルに保存されている
- 主要な取得値がWeb画面の表示と一致する
- APIを再実行せず、保存済みレスポンスを使って加工処理を試せる

### 画面用データに含める情報


### 完了条件

## Phase 4: Next.jsでの表示

1. 企業一覧

### グラフ

### 作業

- 財務指標を年度別に表示する
- 欠損値をゼロとして表示しない

### 完了条件

- 企業一覧から詳細画面へ移動できる
- 直近5年の推移を表とグラフで確認できる

## Phase 5: テストとエラー処理

### テスト対象

- 同じ企業・年度が重複しない
- 年度の並び順が統一される
- 数値のゼロと欠損値を区別できる
- 営業利益率を正しく計算できる
- フリーキャッシュフローの符号を正しく扱える
- 基準年度から3期間前の値が存在しない場合にCAGRを計算しない
- API失敗時に既存の画面用JSONを壊さない
- APIキーがGitの追跡対象に含まれない

### 完了条件

- 一部企業の取得に失敗しても、ほかの企業を表示できる
- 異常なデータを無言で正常値として扱わない
- 主要な計算処理が自動テストで確認される

## MVPの完成条件

- 3〜5社について直近5年分を取得できる
- 主要な財務指標と分析値を表示できる
- 企業詳細と2社比較が利用できる
- 元データ、計算結果、出典を追跡できる
- キャッシュによりEDINET DB APIへのアクセス回数を抑えられる
- API失敗や欠損値があってもアプリ全体が壊れない

## MVPでは対応しないもの

- EDINET公式APIからの直接取得
- XBRLおよびInline XBRLの直接解析
- Arelleの導入
- SQLiteまたはPostgreSQLへの保存
- 全上場企業の取得
- 定期バッチとクラウド運用
- 有価証券報告書本文のAI要約
- 投資スコアと売買推奨
- リアルタイム株価との結合

## 将来の公式EDINET API対応

EDINET DB固有のレスポンスを直接フロントエンドで使用せず、Python側で次の共通形式へ変換する。

```text
企業
年度
標準指標名
値
単位
出典
取得日時
```

将来、データ取得元を公式EDINET APIへ変更する場合は、取得・変換部分のみを交換し、分析処理とNext.js側のデータ形式は維持する。

公式EDINET APIへ移行する際は、次の順に進める。

1. 書類一覧APIから有価証券報告書を特定する
2. `type=5` のCSVを取得する
3. 生のCSVをBronzeデータとして保存する
4. element IDとcontext IDを標準指標へ変換する
5. EDINET DBで取得した値と照合する
6. データ取得元を公式APIへ切り替える

## 推奨する作業単位

最初の実装では、次の範囲だけを扱う。

> EDINET DBから1社の直近5年分を取得し、元レスポンスを保存して、各項目の意味を説明できる状態にする。

この単位が完了してから、分析用JSON、複数企業、画面表示の順に進める。
